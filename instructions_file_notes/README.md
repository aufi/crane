# Migration YAML File Support for `crane transform`

## Overview

This feature adds support for declarative multi-stage transformation configuration via YAML file. Instead of running multiple `crane transform` commands with different flags, you can define your entire transformation pipeline in a single configuration file.

## Benefits

- **Declarative pipeline definition** - Define all transformation stages in one place
- **Reproducible transformations** - Share configuration files across teams
- **Simplified workflow** - Single command to set up complex multi-stage pipelines
- **Version control friendly** - Track transformation configuration alongside code

## File Format

### Minimal Configuration

The simplest configuration only defines the stages:

```yaml
# migration-minimal.yaml
transform:
  stages:
    - name: "10_KubernetesPlugin"
      type: "plugin"
```

### Full Configuration

A complete configuration with all available options:

```yaml
# migration.yaml
transform:
  # Transform parameters (all optional, defaults shown)
  exportDir: "export"
  transformDir: "transform"
  pluginDir: "~/.local/share/crane/plugins"
  kustomizeArgs: ""
  force: false
  
  # Multi-stage pipeline definition
  stages:
    # Plugin stage - automatically regenerated on each run
    - name: "10_KubernetesPlugin"
      type: "plugin"
      description: "Apply default Kubernetes transformations"
      autoRegenerate: true
    
    # Custom stage - protected from automatic overwrites
    - name: "50_CustomLabels"
      type: "custom"
      description: "Add custom labels and annotations"
      autoRegenerate: false
    
    # Additional stages...
    - name: "60_NetworkPolicies"
      type: "custom"
      description: "Add network policies"
      autoRegenerate: false
```

## Stage Types

### Plugin Stages (`type: "plugin"`)

- **Automatically regenerated** on every `crane transform` run
- Stage name should end with `Plugin` suffix (e.g., `10_KubernetesPlugin`)
- Content is generated by crane plugins
- No manual edits needed

### Custom Stages (`type: "custom"`)

- **Protected from overwrites** - require `--force` flag to regenerate
- Stage name should NOT end with `Plugin` suffix (e.g., `50_CustomLabels`)
- Created as empty directories for manual customization
- Users add kustomization patches manually

## Usage Examples

### Basic Usage

```bash
# Create multi-stage pipeline from config
crane transform --config migration.yaml

# Apply transformations (no config needed)
crane apply

# Output is in output/output.yaml
```

### Complete Workflow

```bash
# 1. Export resources from cluster
crane export --context my-cluster

# 2. Create transformation pipeline
crane transform --config migration.yaml
# This creates:
# - transform/10_KubernetesPlugin/
# - transform/50_CustomLabels/
# - transform/60_NetworkPolicies/

# 3. Customize stages (optional)
# Edit transform/50_CustomLabels/kustomization.yaml manually

# 4. Apply transformations
crane apply
# Generates: output/output.yaml
```

### Override Config with CLI Flags

CLI flags take precedence over config file values:

```bash
# Use different transform directory than specified in config
crane transform --config migration.yaml --transform-dir ./my-transform

# Force regeneration of custom stages
crane transform --config migration.yaml --force

# Use different kustomize arguments
crane transform --config migration.yaml --kustomize-args '--enable-helm'
```

### Run Specific Stage Only

Is this needed for single-transform call with instructions file?

```bash
# Run only one stage from the pipeline
crane transform --config migration.yaml --stage 50_CustomLabels
```

## File Locations

Configuration files can be placed anywhere and referenced by path:

```bash
# Local file
crane transform --config migration.yaml

# Absolute path
crane transform --config /path/to/configs/migration.yaml

# Relative path
crane transform --config ../configs/migration.yaml
```

## Configuration Parameters

| Parameter | Description | Default |
|-----------|-------------|---------|
| `exportDir` | Path where kubernetes resources are saved | `export` |
| `transformDir` | Path where transformations are saved | `transform` |
| `pluginDir` | Path where binary plugins are located | `~/.local/share/crane/plugins` |
| `kustomizeArgs` | Additional kubectl kustomize arguments | `""` |
| `force` | Force overwrite of existing stages | `false` |

## Stage Configuration

| Field | Required | Description |
|-------|----------|-------------|
| `name` | Yes | Stage directory name (format: `<number>_<Name>`) |
| `type` | Yes | Stage type: `"plugin"` or `"custom"` |
| `description` | No | Human-readable description of the stage |
| `autoRegenerate` | No | Whether to auto-regenerate (default based on type) |

## Examples

### Two-Stage Pipeline with Custom Labels

```yaml
# migration.yaml
transform:
  stages:
    - name: "10_KubernetesPlugin"
      type: "plugin"
      description: "Default Kubernetes transformations"
    
    - name: "50_CustomLabels"
      type: "custom"
      description: "Add organization-specific labels"
```

After running `crane transform --config migration.yaml`, manually edit:
- `transform/50_CustomLabels/kustomization.yaml` to add your custom patches

### Helm-Enabled Transformation

```yaml
# migration-helm.yaml
transform:
  kustomizeArgs: "--enable-helm --load-restrictor=LoadRestrictionsNone"
  
  stages:
    - name: "10_KubernetesPlugin"
      type: "plugin"
```

### Complex Multi-Stage Pipeline

```yaml
# migration-complex.yaml
transform:
  exportDir: "export"
  transformDir: "transform"
  kustomizeArgs: "--enable-helm"
  
  stages:
    # Stage 1: Base cleanup
    - name: "10_KubernetesPlugin"
      type: "plugin"
      description: "Remove cluster-specific fields"
    
    # Stage 2: Add custom labels
    - name: "20_Labels"
      type: "custom"
      description: "Add team and environment labels"
    
    # Stage 3: Network policies
    - name: "30_NetworkPolicies"
      type: "custom"
      description: "Add network policies for namespace isolation"
    
    # Stage 4: Resource quotas
    - name: "40_ResourceQuotas"
      type: "custom"
      description: "Add resource quotas and limits"
    
    # Stage 5: Security context
    - name: "50_Security"
      type: "custom"
      description: "Enforce security contexts and pod security standards"
```

## Notes

- **`crane apply` does not need the config file** - it automatically discovers stages in the `transform/` directory
- **Stage naming convention**: `<number>_<Name>` where number determines execution order
- **Plugin stages** should end with `Plugin` suffix by convention
- **Custom stages** are created as empty directories - add patches manually
- **Sequential execution**: each stage receives the output from the previous stage
- **Intermediate artifacts**: stored in `transform/<stage>/.work/` for debugging

## Migration from Manual Commands

**Before** (manual multi-stage):
```bash
crane transform
crane transform --stage 50_CustomLabels
crane transform --stage 60_NetworkPolicies
crane apply
```

**After** (with config file):
```bash
crane transform --config migration.yaml
crane apply
```

## Troubleshooting

### Stage already exists

If stages already exist and you want to recreate them:
```bash
crane transform --config migration.yaml --force
```

### Invalid stage name

Stage names must follow the pattern `<number>_<Name>`:
```bash
# Valid
10_KubernetesPlugin
50_CustomLabels

# Invalid
KubernetesPlugin  # missing number
10-KubernetesPlugin  # wrong separator
```

### Config file not found

Ensure the path to the config file is correct:
```bash
# Check if file exists
ls -l migration.yaml

# Use absolute path if needed
crane transform --config $(pwd)/migration.yaml
```
